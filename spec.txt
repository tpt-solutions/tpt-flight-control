## 1. Executive Summary

**TPT Flight Control** is an open-source (Apache 2.0), vertically integrated flight control software platform written in Rust. It is designed to operate across the full spectrum of airborne vehicles—from basic hobby drones to commercial airliners—using a single, unified Rust codebase.

The system achieves this through a **trait-based abstraction architecture** that decouples pure mathematical flight control logic from hardware, operating system, and certification-layer concerns. Conditional compilation via Cargo feature flags enables the same core code to be deployed as a lightweight bare-metal drone firmware or a fully partitioned, DO-178C-certified avionics suite.

TPT Flight Control operates on a **dual-stack model**:
- **TPT Standard:** Compiles against the qualified Ferrocene Rust toolchain with industry-standard RTOS integrations (PikeOS, VxWorks, Zephyr).
- **TPT Sovereign:** Compiles against a fully vertically integrated, formally verified stack including a custom microkernel, verified compiler subset wrapper, and mathematically proven core libraries.

Crucially, TPT includes a comprehensive **Onboard Mapping and GPS-Denied Navigation** subsystem, allowing vehicles to navigate, map, and avoid obstacles in environments where GNSS is degraded, jammed, or entirely non-existent (e.g., urban canyons, indoors, subterranean, or electronic warfare environments).

The initial target market is **electric Vertical Takeoff and Landing (eVTOL) vehicles and Advanced Air Mobility (AAM)**, with a downward-compatible path to consumer drones and an upward-compatible path to commercial aviation via Tier-1 avionics suppliers.

## 2. Purpose & Vision

### 2.1 Problem Statement

The aerospace software industry suffers from:
- **Legacy monoculture:** Flight control systems are predominantly written in C/C++ with codebases dating back 20-40 years.
- **Memory unsafety:** Buffer overflows, use-after-free, and null pointer dereferences remain the dominant class of vulnerabilities in avionics software.
- **GPS Vulnerability:** Modern aviation is overly reliant on GNSS. Jamming, spoofing, and signal degradation in urban environments pose catastrophic risks to autonomous and fly-by-wire systems.
- **eVTOL gap:** Existing avionics architectures were designed for 2-3 actuators and cannot efficiently manage the 8-24+ distributed electric motors of modern eVTOLs.
- **No open standard:** There is no modern, open-source, certification-ready flight control framework that spans from drones to transport-category aircraft.

### 2.2 Vision

TPT Flight Control will become the foundational open-source software platform for the next generation of aviation by:
1. Providing a **mathematically rigorous, memory-safe** flight control core written in Rust.
2. Scaling from a $20 microcontroller in a hobby drone to a $100,000 certified avionics suite in a commercial aircraft.
3. Ensuring **continuous navigation capability** in GPS-denied environments through advanced onboard mapping, Visual-Inertial Odometry (VIO), and Terrain-Aided Navigation (TAN).
4. Enabling **supply chain sovereignty** through the TPT Sovereign stack for defense and national security applications.
5. Accelerating eVTOL and AAM adoption by providing an open, high-performance distributed propulsion controller.

## 3. Design Principles

1. **One Codebase, Many Targets:** All flight control logic exists in a single Rust workspace. Vehicle class and certification level are determined at compile time via feature flags.
2. **`no_std` Core:** The flight control core must compile in a `#![no_std]` environment with zero heap allocation by default. All memory is statically allocated or pool-allocated.
3. **Trait-Based Decoupling:** Every external dependency (hardware, OS, sensor, actuator, spatial map) is accessed exclusively through Rust traits. 
4. **Deterministic Execution:** All control loops execute within bounded, provable time limits. No unbounded loops, no dynamic allocation in hot paths, no panics in flight-critical code.
5. **Fail-Safe & GPS-Independent:** The system must never rely solely on GNSS. If GPS is lost, the system seamlessly transitions to onboard mapping, VIO, or TAN to maintain safe flight or execute a safe landing.
6. **Formal Verification Ready:** The core mathematical libraries are structured to be compatible with formal verification tools (Kani, Creusot, Aeneas) from day one.
7. **Open by Default, Certified by Service:** The source code is Apache 2.0. The certification artifacts and proprietary map databases are the commercial product.

4.2 Execution Model
TPT uses a time-triggered, rate-group execution model:
Rate Group 1 (Inner Loop): 1000 Hz (1ms) — IMU read, attitude estimation, rate control
Rate Group 2 (Outer Loop): 200 Hz (5ms) — Position/velocity control, navigation, VIO update
Rate Group 3 (Guidance & Mapping): 50 Hz (20ms) — Waypoint following, SLAM keyframe insertion, map culling
Rate Group 4 (Telemetry): 10 Hz (100ms) — Logging, telemetry, health monitoring
Rate Group 5 (Background): 1 Hz (1000ms) — Configuration, battery management, diagnostics

5. Core Trait Abstractions
These traits define the contract between the platform-independent core and the platform-specific backends.
5.1 Hardware & Sensor Abstraction Traits
/// Inertial Measurement Unit
pub trait Imu {
    type Error;
    fn read_accelerometer(&mut self) -> Result<Vector3<f64>, Self::Error>;
    fn read_gyroscope(&mut self) -> Result<Vector3<f64>, Self::Error>;
    fn read_magnetometer(&mut self) -> Result<Vector3<f64>, Self::Error>;
}

/// Global Navigation Satellite System
pub trait Gnss {
    type Error;
    fn read_position(&mut self) -> Result<GeoPosition, Self::Error>;
    fn read_velocity(&mut self) -> Result<Vector3<f64>, Self::Error>;
    fn get_fix_type(&self) -> FixType;
    fn is_jammed_or_spoofed(&self) -> bool; // Anti-spoofing integrity check
}

/// Visual / Camera Sensor (for VIO and Mapping)
pub trait VisualSensor {
    type Error;
    fn capture_frame(&mut self, buffer: &mut [u8]) -> Result<FrameMetadata, Self::Error>;
    fn get_intrinsics(&self) -> &CameraIntrinsics;
}

/// LiDAR / Depth Sensor
pub trait LidarSensor {
    type Error;
    fn read_point_cloud(&mut self, buffer: &mut [Point3D]) -> Result<usize, Self::Error>;
}

/// Radar Altimeter (Critical for Terrain-Aided Navigation)
pub trait RadarAltimeter {
    type Error;
    fn read_altitude_agl(&mut self) -> Result<f64, Self::Error>;
}
5.2 Spatial Mapping & Memory Traits
/// Onboard Spatial Map (GPS-Denied Navigation)
pub trait SpatialMap {
    type Error;
    
    /// Insert a new observation into the map
    fn insert_keyframe(&mut self, pose: Pose6DOF, landmarks: &[Landmark]) -> Result<(), Self::Error>;
    
    /// Query the map for obstacles in a given bounding box
    fn query_obstacles(&self, bbox: &BoundingBox) -> Result<Vec<Point3D, MAX_OBSTACLES>, Self::Error>;
    
    /// Get the current estimated pose relative to the map origin
    fn get_local_pose(&self) -> Result<Pose6DOF, Self::Error>;
    
    /// Cull old map data to free memory (sliding window)
    fn cull_distant_data(&mut self, current_pos: Point3D, max_radius: f64);
}

/// Terrain Database (for TAN / TERCOM)
pub trait TerrainDatabase {
    type Error;
    /// Lookup elevation at a given Lat/Lon
    fn get_elevation(&self, lat: f64, lon: f64) -> Result<f64, Self::Error>;
    /// Get a grid of elevations for correlation matching
    fn get_terrain_patch(&self, center: GeoPosition, radius_m: f64) -> Result<TerrainGrid, Self::Error>;
}

6. Flight Control Law Engine
6.1 Control Loop Architecture
The flight control engine uses a cascaded control architecture:
Pilot / Autopilot Command
        │
        ▼
┌─────────────────────┐
│  GUIDANCE LOOP      │  50 Hz  — Waypoint, altitude, heading targets
└────────┬────────────┘
         │ Desired velocities / accelerations
         ▼
┌─────────────────────┐
│  NAVIGATION LOOP    │  200 Hz — Velocity, position hold, approach
└────────┬────────────┘
         │ Desired angles / rates
         ▼
┌─────────────────────┐
│  ATTITUDE LOOP      │  200 Hz — Roll, pitch, yaw angle control
└────────┬────────────┘
         │ Desired angular rates
         ▼
┌─────────────────────┐
│  RATE LOOP          │  1000 Hz — Gyro-rate stabilization (inner loop)
└────────┬────────────┘
         │ Corrective moments / forces
         ▼
┌─────────────────────┐
│  MIXER              │  1000 Hz — Actuator allocation
└─────────────────────┘
6.2 Supported Controller Types
PID: Classical Proportional-Integral-Derivative with anti-windup.
LQR: Linear Quadratic Regulator for optimal multi-variable control.
Adaptive / MRAC: Model Reference Adaptive Control for vehicles with changing dynamics (fuel burn, motor failure).
MPC: Model Predictive Control for constrained optimization (actuator limits, envelope protection, obstacle avoidance).
6.3 Flight Envelope Protection
A hard-coded, non-bypassable envelope protection layer sits between the control laws and the mixer, enforcing maximum pitch/roll, stall protection (AoA), Vne, and minimum energy states.
7. Sensor Fusion Subsystem
7.1 Attitude and Heading Reference System (AHRS)
Algorithms (pluggable via trait):
Complementary Filter: Lightweight, suitable for tpt-micro.
Madgwick / Mahony: Quaternion-based, low computational cost.
Extended Kalman Filter (EKF): Full state estimation. Required for tpt-uas and above.
7.2 GPS-Degraded Fusion Strategy
When GNSS integrity is lost (jamming, spoofing, or signal blockage), the EKF automatically reconfigures its measurement matrix:
Phase 1 (Coast): Rely purely on high-grade IMU dead-reckoning.
Phase 2 (Visual/Depth Aided): Inject Visual-Inertial Odometry (VIO) or LiDAR odometry measurements into the EKF to arrest position drift.
Phase 3 (Terrain Aided): If altitude/terrain features are available, inject Terrain-Aided Navigation (TAN) corrections.
8. Onboard Mapping & GPS-Denied Navigation
This subsystem ensures TPT can operate in "denied" environments (urban canyons, indoors, subterranean, or under electronic warfare).
8.1 Navigation Modalities
Visual-Inertial Odometry (VIO):
Fuses camera frames with IMU data to estimate relative pose.
Uses feature tracking (e.g., FAST/ORB corners) or direct image alignment.
Implementation: Runs on the main flight controller for simple setups, or offloaded to a companion computer (e.g., NVIDIA Jetson) for dense VIO, passing only the pose estimate back to the core via the SpatialMap trait.
LiDAR SLAM (Simultaneous Localization and Mapping):
Uses 3D LiDAR point clouds to build a high-fidelity map of the environment.
Utilizes Iterative Closest Point (ICP) or NDT (Normal Distributions Transform) for scan matching.
Essential for eVTOLs operating in low-visibility (fog, night) or complex 3D urban environments.
Terrain-Aided Navigation (TAN) / TERCOM:
Compares real-time radar altimeter readings and terrain gradients against a pre-loaded Digital Elevation Model (DEM).
Uses particle filters or batch correlation to determine absolute position without GPS.
Highly resilient to jamming; relies on passive radar and stored map data.
8.2 Onboard Map Representation
Maps must be memory-efficient to comply with the no_std / embedded constraints.
Sparse Voxel Octrees (SVO) / Octomaps: Represents 3D space efficiently. Only occupied or "free" space is stored. Ideal for LiDAR SLAM and obstacle avoidance.
Pose Graph / Keyframe Map: For VIO, the map is not a dense 3D grid, but a graph of "keyframes" (camera poses + sparse 3D landmarks). Loop closure algorithms correct drift over time.
Topological Maps: For high-level path planning in GPS-denied areas, the vehicle reduces the metric map to a graph of "nodes" (waypoints/corridors) and "edges" (safe paths), drastically reducing compute requirements for the guidance loop.
8.3 Memory & Compute Management
Sliding Window Optimization: The SLAM/VIO backend only optimizes the last N seconds of data (e.g., a 10-second sliding window). Older keyframes are marginalized out to bound memory usage.
Map Culling: The SpatialMap trait includes a cull_distant_data method. As the vehicle moves forward, the map behind it is deleted from RAM to prevent memory exhaustion.
Companion Compute Offload: For heavy 3D mapping, the tpt-core remains lightweight. A companion computer runs the heavy SLAM algorithms (in C++/Rust) and publishes a lightweight "Local Pose + Obstacle Cloud" to the flight controller over a high-speed internal bus (e.g., Ethernet/UDP or PCIe).
9. Actuator Mixing & Propulsion
9.1 Distributed Electric Propulsion (DEP) Mixer
Translates desired forces and moments into individual motor commands. Includes fault-tolerant reallocation: upon motor failure, automatically recomputes the mixing matrix using pseudo-inverse or quadratic programming to maintain controllability.
9.2 Mixer Strategies
Fixed-wing standard, Multirotor (Quad/Hexa/Octo), Tilt-rotor eVTOL (hover-to-cruise transition), Lift + Cruise, and Vectored thrust.

16. Certification & Compliance Pathway
16.1 Applicable Standards
DO-178C / DO-330 / DO-332 / DO-333: Software certification and formal methods.
DO-326A: Airworthiness security (critical for GPS anti-spoofing and map data integrity).
ARP 4754A / ARP 4761: System safety assessment.
16.2 GPS-Denied Certification Strategy
Certifying a GPS-denied navigation system requires proving that the probability of catastrophic failure due to navigation loss is below 10^-9 per flight hour (for DAL-A). TPT achieves this through:
Dissimilar Redundancy: Using VIO (camera-based) and TAN (radar/terrain-based) as dissimilar backups to GNSS.
Formal Verification: Mathematically proving the bounds of the EKF and TERCOM correlation algorithms.
Integrity Monitoring: Continuous innovation monitoring (checking if the map/visual features match the IMU predictions).

19. Security Model
19.1 Cybersecurity & Anti-Spoofing
GNSS Integrity: Continuous monitoring of GNSS signals for spoofing/jamming (e.g., signal strength anomalies, inconsistent Doppler shifts).
Map Data Integrity: Onboard terrain databases and map data are cryptographically signed to prevent malicious map injection.
Encryption: All communication links use authenticated encryption (ChaCha20-Poly1305).
19.2 Supply Chain Security
All Rust dependencies vendored and audited. SBOM generated for every release. Reproducible builds.
20. Open Source Governance
License: Apache License 2.0 for all source code and reference hardware.
Commercial Products: DO-178C data packages, proprietary high-fidelity terrain databases, and certified companion-compute SLAM binaries.
Contribution: DCO sign-off required. Flight-critical and mapping-critical code requires domain-expert review.
